# **總則**

<a id="_Toc233823595"></a>
## **計畫目標**

本案係財團法人國營臺灣鐵路股份有限公司職工福利委員會為提升全臺職工福利補助服務品質，因應補助申請文件紙本化、審核流程分散、撥款與傳票作業需人工彙整、歷程查詢不易及稽核追蹤成本較高等現況，規劃建置「職工福利補助申請暨管理系統」。

本系統以補助案件全生命週期數位化為核心，整合職工端申請、管理端審核、財務傳票產製、AI OCR 輔助辨識、特約商店查詢、訊息推播、權限控管及稽核留痕等功能，建立申請、審核、撥款、領款、傳票、歸檔及稽核之一體化作業閉環。

本案推動目標包含：

1. 建置全會統一之福利補助申請平台，降低紙本傳遞與人工彙整成本；
2. 以電子化審核與多階簽核機制提升案件處理效率；
3. 導入 AI OCR 輔助辨識，協助承辦快速檢視附件內容與異常資料；
4. 建立財務傳票自動產製及管理機制，提升帳務作業效率；
5. 建立角色權限、操作日誌與資安控管，確保個人資料與補助案件歷程可追溯、可稽核、可管理。

- 職工端（跨平台 PWA 智慧申請服務系統）：登入機制、補助申請、申請進度查詢、文件上傳、特約商店瀏覽、訊息接收、個人資料維護、補助歷程查詢。
- 管理端（核心行政與核准管理系統）：組織與人員權限管理、補助案件審核、補助項目設定、代申請與禮金申請機制、財務傳票自動產製、訊息發布、特約商店管理、統計報表、資料匯入與對比模組。

# **系統架構與技術規劃**

<a id="_Toc233823602"></a>
## **整體架構概述**

本案系統採可擴充、高可用之雲端架構，以 RESTful API 服務層為核心，模組化設計，區分「使用者入口」、「應用服務層」與「雲端基礎設施」三層，支援現代化架構與資安防護，並滿足政府資安 C 級與業務永續運作需求。

三層架構說明如下：

- **使用者入口層**：含職工端 PWA（跨平台漸進式網路應用）、管理端後台、稽核與系統後台，提供跨電腦／手機之響應式操作介面。
- **應用服務層**：以 RESTful API 提供身分驗證與 RBAC、補助申請與審核、AI OCR 輔助辨識、財務傳票產製、訊息通知、特約商店管理、操作日誌與稽核等服務模組。
- **雲端基礎設施層**：含 SQL 資料庫、物件儲存（Object Storage）、WAF、流量監控（Monitoring）及備份與災難復原（Backup／Disaster Recovery）。

<a id="_Toc233823603"></a>
## **邏輯架構**

系統以前後端分離設計，前端採 PWA 免上架部署、響應式介面；後端以 RESTful API 為唯一服務介面，業務邏輯集中於應用服務層，確保前端跨平台一致與後端可獨立擴充。各功能模組透過 API 介接，降低耦合並利於維運與測試。身分驗證、權限控管與操作日誌為橫切服務，貫穿各業務模組。

<a id="_Toc233823604"></a>
## **部署架構**

系統部署於公有雲，依需求說明書第九點雲端伺服器規格辦理，具備處理高頻率影像上傳、AI 辨識及長達 3 年法定資料保存之能力。資料庫採高速存取區，附件影像置於物件儲存，前端經 WAF／CDN 保護與加速，並以監控服務掌握系統健康度。備份採每日自動備份、異地備援與災難復原規劃，並定期執行備份還原測試。

### **登入與身分驗證模組**

1. **功能概述**

本模組提供職工端與管理端之身分識別、登入驗證、密碼救援及角色權限控管。職工端採員工編號作為帳號配合身分證以及 Email 發送的一次性驗證碼（OTP）進行第一次登入；登入後進行密碼設定與資料填寫。管理端或忘記密碼管理於每次登入都須使用 OTP。系統依角色權限控管機制區分六大角色，確保使用者僅能存取其職責範圍內之功能與資料。

1. **作業流程**

- 職工以臺鐵員編作為帳號、身分證字號作為密碼，發起第一次登入；系統寄送 OTP 一次性驗證碼至該信箱。
- 使用者於時效內輸入 OTP，系統驗證通過後建立登入工作階段（Session）。
- 忘記密碼採三步驟救援：身分確認（Email）→ 寄送驗證連結／OTP → 重設密碼。
- 管理端使用者於 OTP 通過後，即完成登入並進入系統。
- 系統依使用者角色載入對應之功能選單與資料範圍，並寫入登入紀錄。

![](images/4309381362a506cc26f98f0b8c57b577eacc6ff8935eaed1397cb535d5be3fc3.png)



圖 5 身分驗證與忘記密碼三步驟流程圖


1. **系統處理邏輯**

- 關鍵存取節點（如登入、密碼重設等）導入人機驗證（CAPTCHA 或行為檢測）機制，有效防範自動化腳本與暴力破解攻擊。
- OTP 具時效與重送節流（throttling）控制，連續錯誤達門檻即暫時鎖定並記錄。
- 密碼以單向雜湊加鹽（salted hash）儲存，傳輸全程採 TLS 1\.2／1\.3 加密。
- RBAC 以「角色—權限—資料範圍」三層模型運作，權限異動即時生效並留存軌跡。
- 工作階段具閒置逾時自動登出機制，降低未授權存取風險。

1. **使用者操作步驟**

1. 開啟系統 → 輸入 Email → 點選「取得驗證碼」。
2. 至信箱取得 OTP → 輸入驗證碼 → 登入。
3. 如忘記密碼，點選「忘記密碼」依三步驟完成重設。

1. **資料流向**

登入請求（Email）→ 驗證服務產生 OTP → 經 Email 介接（臺鐵 Outlook）送達使用者 → 使用者回填 OTP → 驗證服務比對 → 核發 Session 並將登入事件寫入操作日誌（保存 ≥ 3 年、防竄改）。密碼與 OTP 不以明文留存或落地。

1. **例外處理機制**

- OTP 逾時或錯誤：提示重新取得，連續錯誤逾門檻暫時鎖定並記錄事件。
- Email 寄送失敗：提供重送，並於後台告警以利機關協助排查介接。
