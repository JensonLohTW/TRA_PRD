本文件是 26 份子 PRD 的總控索引，用來對齊：

- 模塊清單是否完整
- 模塊命名與邊界是否一致

## 4. 核心主鏈

### 4.1 補助與發款主鏈

職工登入並完成資料完整性檢查 → 建立補助申請 → 上傳附件與數位切結 → 產製 PDF → 實體核章 → 主管審核 → 核准進待發款池 → 建立發款批次 → 主管核准批次 → 人工撥款回填 → 職工領款確認 / 異議。

### 4.2 公告主鏈

公告草稿 → 設定投放對象 → 設定可見窗口與排程 → 送審 → 核准後發布 → 前台公告中心與下載專區展示 → 觸及/瀏覽追蹤。

### 4.3 商店主鏈

商店建檔 → 合約送審 → 核准且在有效期間內 → 前台列表+地圖展示 → 到期排程下架。

### 4.4 治理主鏈

登入/授權/檔案/通知/審批事件 → 稽核日誌 → 02:00 掃描 → 必要時告警 → 資安工作台處置 → `.log.gz` 封存與報表。

## 5. 建議研發排期

### Phase 0

M07、M08、M09、M23 最小骨架。

先立字典、檔案、通知、稽核事件模型，否則後續模塊會反覆返工。

### Phase 1

M01～M06。

先立身份、組織、權限、員工與資格底座，因為 Portal / Admin / Security Console 都要先經過這一層。

### Phase 2

M10、M11、M12。

先把固定審批鏈、待辦與超時事件打通，供 BEN/PAY/ANN/MCH 共用。

### Phase 3

M13、M14、M15、M26。

先打通補助申請主鏈與 AI 輔助，但 AI 僅作輔助，不可阻斷客戶核心流程。

### Phase 4

M16、M17、M18。

完成待發款池、建批、人工回填、領款確認與異議處理。

### Phase 5

M19、M20、M21、M22。

完成公告與商店兩條「審批後前台展示」鏈。

### Phase 6

M24 與 M23 完整化。

把資安工作台、封存報表與處置流程收口。

## 6. 原始需求對齊提醒

- 不是 24 份，而是 26 份文件。
- 不是 Local-first 登入，而是正式同仁 Graph 驗證、部分工時內部帳號。
- 不是四類補助，而是以原始 PRD 的 6 大項 13 種補助為主。
- 不是把 WF 做成通用 BPM 平台，而是先落實客戶明示的行政流程。
- 商店前台受眾以登入職工為主，且必須提供列表+地圖與前台保護機制。
- SEC 留存應以最近 30 天熱查、至少 3 年封存為準，不應回寫成 6 個月 / 1 年 / 3 至 5 年的錯誤基線。
