- 平台定位

## 1. 平台定位

臺鐵職工福利平台不是單一補助申請系統，而是把補助申請、審批、發款、公告、特約商店、組織權限與稽核資安，放進同一條可追蹤、可治理的 B2E 內部平台。

對職工，重點是好懂、可查、有通知、有結果。
對承辦與主管，重點是可審、可退回、可追蹤、可責任切分。
對系統管理與資安，重點是可配置、可控權限、可稽核、可封存。

## 4. 平台主鏈

### 4.1 補助鏈

登入 → 個資完整性檢查 → 補助申請 → 數位切結 → PDF 產製 → 實體核章 → 審核 → 核准 / 退回 / 駁回。

### 4.2 發款鏈

核准案件入待發款池 → 建立發款批次 → 主管核准 → 人工撥款回填 → 領款確認 / 異議。

### 4.3 公告鏈

公告草稿 → 投放對象 → 可見窗口與排程 → 送審 → 核准發布 → 前台公告中心 → 觸及/瀏覽追蹤。

### 4.4 商店鏈

商店建檔 → 合約送審 → 核准且有效 → 前台列表+地圖展示 → 到期下架。

### 4.5 治理鏈

登入/授權/檔案/流程/下載等事件 → 稽核日誌 → 掃描 → 告警 → 資安處置 → 封存與報告。

## 6. MVP 對齊原則

- 正式同仁以 Microsoft Graph API 驗證為主，部分工時人員使用內部帳號。
- 啟活與重設以 6 碼 OTP 為主，並區分 Outlook / 個人 Email。
- 補助主鏈必須保留數位登錄、PDF 產製與實體核章斷點。
- 發款採人工回填，不納入銀行 API。
- 公告與商店都需先審批，再進前台展示。
- 商店前台需支援列表+地圖與前台保護措施。
- SEC 以最近 30 天熱查、至少 3 年封存、02:00 掃描、`.log.gz` 封存為基線。

## 7. 總驗收口徑

最終驗收以鏈路打通為主，不以頁面數量為主：

1. 職工能完成補助申請到領款確認的完整主流程。
2. 承辦能完成建批、送審、人工回填與異議處理。
3. 公告能完成草稿、送審、發布與前台展示。
4. 商店能完成合約審批、前台展示與到期下架。
5. 權限與資料範圍能正確限制列表與詳情。
6. 高風險操作可被稽核、告警、查詢與封存。
